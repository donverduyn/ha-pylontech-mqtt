## Automation / CI (4, 5, 6, 16)

4. **Release automation** — Optimize release workflows; auto-versioning via
   semantic versioning with conventional commits. Current flow
   (`.github/workflows/autorelease.yaml` → `release.yaml`) is manual:
   a PR bumps `manifest.json`'s `version` field by hand (rc suffix for
   prereleases, e.g. `2.1.0-rc1`), and merging to master triggers the
   publish pipeline off that diff. `release.yaml` itself runs no tests —
   it relies on every commit on master already having passed the full
   unit → image-build/smoke-test → e2e gauntlet on its PR. Switching to
   conventional-commit-driven version bumps would replace the manual
   manifest-bump step but needs to preserve the deliberate-release-as-PR
   gate that currently holds finals behind an rc phase (see
   `CONTRIBUTING.md`'s hardware-validation gate).

5. **Device response capture** — Add a Make command + scripts to automate
   retrieving responses from real devices. `scripts/capture_transcript.py`
   already exists and does exactly this (connects like `src/main.py`

   redacts the `Barcode` field). What's missing is a `make` target wrapping
   it, and — per `tests/fixtures/transcripts/README.md` — an actual
   contributed transcript: the directory is empty, tracked as a P0 gap
   (the stub is hand-authored from documented protocol, never validated
   against real firmware).

6. **Stub server improvement** — Use the captured responses to automatically
